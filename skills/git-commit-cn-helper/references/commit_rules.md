# Git 提交规范

## 提交格式

```text
<图标> <类型>([范围-可选]): <中文描述>

- [中文描述1-可选]
- [中文描述2-可选]
- [中文描述3-可选]
```

## 分类

- `🎉 init:` 初始化仓库、脚手架、基础模块或首次落地的新子系统
- `✨ feat:` 新功能、新命令、新配置能力或用户可感知的新行为
- `🐞 fix:` 修复 Bug、错误逻辑、兼容性问题、崩溃或异常行为
- `📃 docs:` 文档变更、注释修改、README 更新、示例说明补充
- `🌈 style:` 代码格式调整、排版统一、lint 自动修复，不影响功能
- `🦄 refactor:` 代码重构或结构整理，不新增功能，不修复 Bug，也不以性能提升为主要目标
- `🎈 perf:` 明确以性能优化为目标的改动，例如减少耗时、内存占用或重复计算
- `🧪 test:` 新增测试、修改现有测试、补充回归用例或测试基建
- `🔧 build:` 构建系统、打包流程、编译配置、发布产物或依赖构建链路变更
- `🐎 ci:` CI/CD 工作流、流水线、自动化检查、发布作业或托管平台配置
- `🐳 chore:` 日常维护类杂项，不属于 `build` 或 `ci`，例如通用脚本、仓库整理、辅助配置
- `↩ revert:` 回滚既有提交或撤销某项历史变更

## 选择建议

- 如果一次改动跨越多类，优先选择最能体现主要目的的类型。
- 推荐优先级：`revert` > `feat` / `fix` / `perf` > `refactor` > `test` > `docs` / `style` > `build` / `ci` / `chore`。
- 仅当“性能提升”是核心目标时使用 `perf`；仅做结构整理时使用 `refactor`。
- `build` 面向本地构建与产物链路，`ci` 面向平台自动化流程，剩余维护项再归入 `chore`。

## 执行要求

- 先分析 `git --no-pager diff --cached --text`，不要猜测未暂存内容。
- 如有必要，先看 `git --no-pager log -n 5 --oneline`，尽量贴近近期提交风格。
- 描述需准确概括改动核心，避免使用模糊词。
- 最终执行 `git commit` 前必须取得用户确认。
